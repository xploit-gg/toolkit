Pentest Reporting

Structured report writing with CVSS scoring, proof-of-concept evidence, and actionable remediation recommendations.

📋 Report Structure

Standard Pentest Report Sections Reference 
1. Cover Page
   - Engagement title, date, client name, classification (Confidential)

2. Executive Summary
   - High-level overview for non-technical stakeholders
   - Overall risk rating, key findings summary
   - Business impact in plain language

3. Scope & Methodology
   - In-scope targets (IPs, domains, apps)
   - Out-of-scope items
   - Testing methodology (black box / grey box / white box)
   - Tools used
   - Testing dates and tester name

4. Findings Summary Table
   - Finding name | Severity | CVSS Score | Status

5. Detailed Findings
   - One section per vulnerability (see Writing a Finding below)

6. Remediation Summary
   - Prioritized list of fixes

7. Appendix
   - Raw tool output, screenshots, raw scan logs

📊 CVSS Scoring

CVSS v3.1 Base Metrics CVSS 
# Attack Vector (AV)
N = Network   | A = Adjacent | L = Local | P = Physical

# Attack Complexity (AC)
L = Low | H = High

# Privileges Required (PR)
N = None | L = Low | H = High

# User Interaction (UI)
N = None | R = Required

# Scope (S)
U = Unchanged | C = Changed

# Impact — Confidentiality / Integrity / Availability
N = None | L = Low | H = High

# Score ranges
0.0       → None
0.1–3.9   → Low
4.0–6.9   → Medium
7.0–8.9   → High
9.0–10.0  → Critical

# Calculator: https://nvd.nist.gov/vuln-metrics/cvss/v3-calculator

🐛 Writing a Finding

Finding Template 
Finding Title:   SQL Injection in Login Parameter
Severity:        Critical
CVSS Score:      9.8 (AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H)
Affected URL:    https://target.com/login
Parameter:       username

─────────────────────────────────────────
Description:
  The login endpoint is vulnerable to SQL injection via the
  'username' parameter. Input is passed directly to the SQL
  query without sanitization or parameterized queries.

─────────────────────────────────────────
Proof of Concept:
  Request:
    POST /login HTTP/1.1
    Host: target.com
    Content-Type: application/x-www-form-urlencoded

    username=admin'--&password=anything

  Response: HTTP 200 OK — logged in as admin

  [Screenshot of successful bypass here]

─────────────────────────────────────────
Impact:
  An attacker can bypass authentication, extract all database
  contents including user credentials, and potentially execute
  operating system commands.

─────────────────────────────────────────
Remediation:
  1. Use parameterized queries / prepared statements.
  2. Implement input validation and whitelist allowed characters.
  3. Apply principle of least privilege to database accounts.
  4. Deploy a Web Application Firewall (WAF) as a secondary layer.

🚦 Severity Ratings

Severity Quick Reference Reference 
CRITICAL  → Remote code execution, full system compromise, auth bypass
           Examples: SQLi with RCE, unauthenticated RCE, critical CVE

HIGH      → Significant data exposure, privilege escalation, auth bypass
           Examples: SQLi (data dump), stored XSS, IDOR on sensitive data

MEDIUM    → Limited data exposure, needs user interaction, partial access
           Examples: Reflected XSS, IDOR on non-sensitive data, LFI

LOW       → Minimal impact, informational value
           Examples: Verbose errors, version disclosure, missing headers

INFORMATIONAL → Best practice issues, no direct exploitation
           Examples: Missing security headers, SSL config weaknesses
